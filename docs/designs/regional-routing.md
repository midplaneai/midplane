# Regional Neon Routing — EU/US Physical Isolation

Branch: `lange-labs/regional-neon-routing` | Generated by /plan-ceo-review + /plan-eng-review on 2026-05-19, rebased on PR #29

## Problem

App layer and schema enforce regional partitioning (immutable `customers.region`, composite FKs, per-region KMS keys, region-aware spawner), but the **control-plane web app** runs against a **single Neon database in `eu-central-1`**. US customers' control-plane rows (orgs, encrypted DSN ciphertext, audit index) physically live in the EU. Closes that gap.

The data-plane is already region-correct (`eu.midplane.ai` / `us.midplane.ai` MCP runtime apps). The gap is exclusively the control-plane Postgres.

Key evidence: `packages/db/src/index.ts:11` (single `DATABASE_URL`); `fly-web.toml:36` (`Neon prod branch (eu-central-1)`); no `fly-web-us.toml` exists.

## Approach

**Two control-plane apps, two Neon projects.** `midplane-web` (EU, fra) reads `DATABASE_URL_EU` only. New `midplane-web-us` (iad) reads `DATABASE_URL_US` only. Each app is dumb — one DB, one KMS key, one region. `app.midplane.ai` CNAMEs to `eu.app.midplane.ai`; the EU app does apex double-duty (auth → 302 to regional subdomain; unauth → existing `/signup/region` picker).

Tenant identity is `clerkOrgId` (per PR #29). Region lives in **organization** `publicMetadata.region`, read from the JWT in middleware. Bounded staleness is acceptable — if a stale JWT hits the wrong region, middleware redirects.

Rejected: one-app-two-DBs (cross-region-write risk), defer-with-disclosure (closes US enterprise door, forces painful future migration).

## V1 Data Migration Stance

**V1 routes new signups only.** Existing US-tagged rows stay in EU Neon until a V2 migration. The heads-up email tells these customers: their data (and ongoing audit events, which keep landing in EU Neon via the EU control plane's indexer) stays in EU until they ask to migrate. New signups today route to US. V2 (in TODOs): dump + re-encrypt DSN ciphertext under US KMS + load + delete + verify.

## Scope

1. **Provision US Neon project** in `us-east-2`. Two projects total.
2. **Add `fly-web-us.toml`** for `midplane-web-us` in `iad`. Rename `fly-web.toml` → `fly-web-eu.toml` (cosmetic; Fly app name and existing TLS unchanged). Mirror the EU comment block including `MIDPLANE_OSS_IMAGE` for parity.
3. **Refactor `packages/db/src/index.ts` `getDb()`** to require a region argument; read `DATABASE_URL_EU` / `DATABASE_URL_US`. Use a `Map<Region, Db>` cache (single-slot would let one region win the slot; no cache would leak a 10-connection pool per call). **Env-var locality:** each regional app only sets its own `DATABASE_URL_<REGION>`; the other is unset, so a cross-region `getDb()` call throws at the env-var read. Add `MIDPLANE_REGION="eu"|"us"` to each app for module-init callers (specifically `apps/web/src/lib/mcp-proxy.ts:83` constructs the proxy context before any request scope exists). Migrate ~15 callsites: request-scope callers read region from a new `apps/web/src/lib/region-context.ts` AsyncLocalStorage; module-init callers read `MIDPLANE_REGION`.
4. **Region resolution middleware** extends the existing `apps/web/src/middleware.ts` (today: just `clerkMiddleware` + `auth.protect()`). Apex-host short-circuit runs BEFORE `auth.protect()` so apex unauth lands on `/signup/region`. After `auth.protect()`: read `auth().orgId` for tenant, `auth().sessionClaims.org?.publicMetadata?.region` for region (org-scoped, not user); set ALS context; redirect cross-region (US user on EU app → 302 to US). Null orgId or null region → redirect to `/signup/region`. Signup writes always enforce (item 5); only this middleware redirect path has a `MIDDLEWARE_ENFORCE` flag — flipped to true after the first hour of monitoring shows zero `region.routing_anomaly` alerts.
5. **Signup writes Clerk org metadata.** Extend `apps/web/src/lib/customer.ts` `upsertCustomerRegion(region)` to write atomically: DB `customers` row via the existing `onConflictDoNothing` path (keyed off `clerkOrgId`), then `clerkClient.organizations.updateOrganizationMetadata(orgId, { publicMetadata: { region } })`, awaited before return. If the Clerk write fails after the DB write, the customer row is consistent but the org lacks metadata — backfill (item 6) catches it.
6. **Backfill script `scripts/backfill-clerk-region.ts`** — run pre-deploy with `--write` flag against prod Clerk + prod EU Neon. Iterates Clerk **organizations** via `clerkClient.organizations.getOrganizationList({ limit: 100 })` with pagination; joins `customers` by `clerk_org_id` to read the region; writes org metadata. Idempotent (no-op if already correct). Audit row per org via `emitConfigAuditRow` with `REGION_CHANGED` event (item 7).
7. **Staff endpoint `POST /admin/customer/:id/region`** behind env-var allowlist `MIDPLANE_STAFF_USER_IDS` (comma-separated Clerk user IDs; Midplane staff are not customer org members, so org-role doesn't apply). Audit log via existing `emitConfigAuditRow` helper. Requires migration `0016_region_changed_event.sql` (DROP+ADD CHECK constraint on `audit_events_index.event_type`, mirroring 0015's pattern), widening the TS event-type union in `apps/web/src/lib/connections.ts:47`, adding to the inline policy-event IN clauses in `apps/web/src/lib/audit.ts:305` and `:644`, adding to `event-badge.tsx` and `VALID_EVENT_TYPES` in `packages/router/src/indexer.ts:117-123`. Local-region only in V1; rate-limited 10/hr/staff. Best-effort `revokeSession` enumeration across the target org's members (no atomic org-wide revoke exists; race window acceptable for a rare audited action).
8. **DNS + TLS:** `eu.app.midplane.ai` CNAME → `midplane-web.fly.dev`; `us.app.midplane.ai` CNAME → `midplane-web-us.fly.dev`; `app.midplane.ai` CNAME → `eu.app.midplane.ai`. Fly matches certs by SNI, so **three certs** required: `fly certs add eu.app.midplane.ai --app midplane-web`, `fly certs add us.app.midplane.ai --app midplane-web-us`, `fly certs add app.midplane.ai --app midplane-web`. Wait for `fly certs check` green on each before cutover.
9. **Region picker reuses `/signup/region`.** No new route. Extend the existing page to render for unauth apex requests too (today it `redirect("/")` if no userId). Copy: "Pick where your data should live. This decision is permanent." Two cards (EU/US), keyboard-navigable, focus ring, stacks at 320px, uses existing `Card`/`Button` primitives. On submit, sets `__Host-midplane-region` cookie (`Secure`, `SameSite=Lax`, `Path=/`, `Domain=.midplane.ai`, 1-year) as a routing hint; Clerk session always wins.
10. **Indexer per region.** The indexer runs in the **control-plane** web app (`apps/web/src/lib/mcp-proxy.ts` instantiates it), not the data-plane runtime apps. Each regional control-plane app's indexer instance writes to its regional Neon via the `MIDPLANE_REGION`-driven `getDb()` call. No code change in `packages/router/src/indexer.ts`.
11. **Cross-region secret cleanup.** Each regional app drops the OTHER region's secrets so the spawner can't accidentally route a connection through the wrong control plane. EU drops `FLY_APP_US`, `FLY_REGION_US`, `MIDPLANE_KMS_KEY_US`, `MIDPLANE_KMS_DEV_KEY_US`, `MIDPLANE_PUBLIC_HOST_US`, `DATABASE_URL_US`. US symmetrically drops EU equivalents.
12. **Region badge in topbar.** Semantic color tokens `region.eu` / `region.us` (add to `globals.css` + `tailwind.config.ts`). Display-only, `aria-label="Your data is hosted in <region>"`. Visible on mobile.
13. **`/api/health` enriched** to report region identity, Neon round-trip (`SELECT 1`), and indexer lag (`now() - max(audit_events_index.created_at)`). Fly's existing http_service health check polls this every 30s.
14. **Structured log middleware** injects `region` tag on every server log line (Next.js `instrumentation.ts` or per-route wrapper).
15. **Migration scripts:** `migrate:eu`, `migrate:us`, `migrate:all` in package.json — each invokes `packages/db/src/migrate.ts` against the region's `DATABASE_URL`. No CI gate; `migrate:all` solves the "forgot one region" case.
16. **One alert: `region.routing_anomaly`** — fires on null `publicMetadata.region` for an authenticated session, cross-region-redirect rate > 1%, or `/api/health` 503. One alert with three triggers beats three separately-routed alerts at this scale.

## Rollback

1. **DNS cutback** — point `us.app.midplane.ai` CNAME at `midplane-web` (EU). US users hit EU app; middleware redirects them; they land on `eu.app.midplane.ai` reading their (still-EU-resident) data. Incident-only; not steady state.
2. **Middleware enforce flag** — `MIDDLEWARE_ENFORCE=false` reverts to log-only without redeploy.
3. **Backfill rollback** — delete `publicMetadata.region` via Clerk (enumerated from backfill audit log). Do not roll back after new signups have been written under the new system.
4. **US app rollback** — `fly apps destroy midplane-web-us`. DNS cutback is the user-visible step.

Rollback drill before launch: in staging, set `DATABASE_URL_US` to an invalid host, verify DNS cutback lands all traffic on EU within 60s.

## Out of scope

Cross-region disaster failover (separate problem; residency-violating if done casually); multi-region application caching (no caching layer today); real-time cross-region replication (defeats residency); additional regions.

## Key Engineering Decisions

1. Two physical Neon projects in `eu-central-1` and `us-east-2`. Two projects in one Neon org is sufficient — residency requires "data at rest in `us-east-2`", which one Neon project there satisfies.
2. DNS-based subdomain routing — surfaces residency in the URL bar; cheap.
3. Single Clerk instance; region in **organization** `publicMetadata`; read from JWT with bounded staleness.
4. No Cloudflare Worker. Apex routes via EU app middleware.
5. `getDb(region)` is required; `Map<Region, Db>` cache; env-var locality is the cross-region-write guard.
6. `MIDPLANE_REGION` env var serves module-init callers; AsyncLocalStorage serves request-scope callers.
7. Region pinning enforced at request middleware — cross-region redirect, never silent allow.
8. V1 ships routing, not historical migration. Existing US rows stay in EU until V2.
